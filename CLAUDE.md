# CLAUDE.md

Project instructions for Claude Code.

> **SPEC.md** is the single source of truth for worldbuilding, UX conventions, mechanics, and data tables. It is fully auto-generated by `scripts/build-spec.sh` (7 sections: Vision → Setting → Mechanics → UX → Devlog → Plan → Data). Authored content lives in `scripts/templates/` — edit those, not SPEC.md directly.
>
> **For any UI or graphics work:** Read the UX section of `@SPEC.md` before writing rendering code. The aesthetic rules (colors, typography, layout patterns, world-space vs HUD placement, component conventions) are all there.
>
> **For any lore or worldbuilding work:** Read the Setting section of `@SPEC.md` for faction descriptions, location names, and world tone.

## Commands

- **Dev server:** `npm run dev` (Vite, hot-reload)
- **Build:** `npm run build` (output to `dist/`)
- **Preview build:** `npm run preview`
- **Lint:** `npm run lint` (ESLint)
- **Type check:** `npm run check` (TypeScript `checkJs`, no emit)
- **Both:** `npm run validate` (lint + check)

### Validate After Every Change

**MANDATORY:** After completing any feature, fix, or unit of work, run `npm run validate` before considering the work done. Fix any new errors or warnings your changes introduced. Do not leave lint violations or type errors behind — the codebase should stay clean after every change.

## Feature Code Workflow

All features move through two stages:

1. **PLAN.md** — feature concepts with a letter code (e.g. `AN`). Ideas start rough and get refined here. Coded items are ready to build directly from this file.
2. **DEVLOG.md** — one line appended when the feature ships. Code is retired here permanently.

Small tweaks and bug fixes live in **FIXES.md** (no codes, just a flat bullet list).

**Assigning codes:** Check PLAN.md for the "next available code" header. Codes are sequential two-letter suffixes after `AM` (the last DEVLOG entry): `AN`, `AO`, `AP`, etc. Assign the next available code to each new idea when you add it to PLAN.md.

Two harnesses. Both run on the same `startLoop` — each implements `update(dt)` / `render()`.

| URL | Mode | Purpose |

| `designer.html` | Unified Designer | Browse all ships, POIs, and weapons with stats panels |

- **Maps:** `data/maps/` — each file exports `MAP`; pass `?map=<name>` to select

**Available maps:**

| Param | File | Purpose |

**Every development iteration**, update the relevant map in `data/maps/` to include new entities/features so they're easy to reach. Tell the user to open `editor.html?map=<name>` to validate.

### `editor.html` — DOM Panels

Editor UI chrome is DOM-based (not canvas). Classes in `engine/ui/editorPanels.js`, styled by `css/editor.css`:

- **`EditorSidebar`** — right panel for object browsing/placement (toggle: `-` key)
- **`EditorItemMenu`** — left panel for adding items to cargo (toggle: `[` key)
- **`EditorPanBanner`** — pan mode indicator banner

Only `_renderDebugOverlay()` remains on canvas (world-space entity tracking).

### `designer.html` — Unified Designer

- **Source:** `engine/test/designer.js`, entry: `engine/designer-main.js`
- **Navigation:** `↑/↓` change category, `←/→` cycle item, `T` toggle rotation (ships), `R` reset view, scroll/drag to zoom/pan
- **Deep-link:** `designer.html?category=<cat>&id=<slug>`
- **In scope:** `data/hulls/**`, `engine/entities/**`, `engine/modules/**`, `engine/rendering/colors.js`, `data/ships/**`, `data/locations/**`

### `designer.html` — DOM Panel

Stats panel is DOM-based. `DesignerPanel` class in `engine/ui/designerPanel.js`, styled by `css/designer.css`. Preview renderers (ship silhouettes, grids, module icons, weapon animations, module slot boxes + connection lines) remain on canvas.

## Architecture

### Entry Point

`index.html` → `engine/main.js` → creates `GameManager` → starts game loop.

### Core Systems

- **`engine/game.js` / `GameManager`** — central orchestrator; owns entities, camera, renderer, HUD, particle pool, subsystems; drives `update(dt)` and `render()`
- **`engine/systems/playerInventory.js` / `PlayerInventory`** — owns scrap, fuel, cargo, modules, weapons, ammo. GameManager exposes forwarding accessors (`game.scrap`, etc.)
- **`engine/systems/` subsystems** — `SalvageSystem`, `RepairSystem`, `CollisionSystem`, `BountySystem`, `WeaponSystem`, `InteractionSystem`, `NavigationSystem`. Each owns its domain state and exposes `start()`/`update()`/`cancel()` as appropriate.
- **`engine/loop.js`** — fixed-timestep loop (60 ticks/sec)
- **`engine/camera.js`** — world↔screen transform, exponential-lerp follow, visibility culling
- **`engine/input.js`** — keyboard hold/just-pressed, mouse position/buttons, flushed each tick
- **`engine/renderer.js`** — clears canvas, draws starfield, renders entities, then HUD/UI overlays
- **`engine/hud.js`** — thin orchestrator; bottom strip is DOM (`#hud-bottom`); canvas sub-renderers in `engine/hud/`. See `@SPEC.md` UX section for layout details.

### Entity Types

`Entity` base class (`engine/entities/entity.js`). Every entity has `entityType`: `'ship'`, `'station'`, `'projectile'`, `'loot'`, `'explosion'`, `'entity'`. **Never use `instanceof`** — use `entity.entityType === 'ship'` (lint enforced). Types: `Ship` (armor/hull/weapons), `Projectile`, `LootDrop`, `Particle`, `Station`, `Planet`. Derelicts are Ships with `crew = 0` (`ship.isDerelict` getter).

**Character** (`engine/entities/character.js`) — a person who inhabits a ship. `boardShip(ship)` sets `ship.captain`; ship delegates faction/relation/ai to the character via getters. Concord machines are unmanned (no Character).

**Factories** (`engine/entities/registry.js`): `createNPC(characterId, x, y)` for manned ships, `createShip(shipId, x, y)` for unmanned/Concord, `createDerelict(data)` for wrecks. Hull classes in `data/hulls/*/hull.js`; Concord entity subclasses in `engine/entities/concord/`.

### Key Patterns

- **Entity list** — all entities in `GameManager.entities[]`, updated/rendered polymorphically; inactive purged each tick
- **Collision detection** — projectile-vs-ship circle checks in `CollisionSystem.update()`
- **Enemy AI** — `engine/ai/shipAI.js`; home position + patrol; aggro/deaggro range; behaviors set via `this.ai = { ...AI_TEMPLATES.X }` from `@data/index.js`: stalker, kiter, standoff, lurker, flee. All AI runtime state lives on `ship.ai.*` (e.g. `ship.ai._aggro`, `ship.ai._patrolAngle`, `ship.ai._lurkerState`). The ship's AI status string is `ship.aiStatus` (not `aiState`).
- **Neutral AI** — `engine/ai/shipAI.js`; dispatches on `ship.ai.passiveBehavior` ('trader' or 'militia'). Trade route fields: `ship.ai._tradeRouteA/B`. Orbit fields: `ship.ai._orbitCenter/Radius/Speed/Angle`.
- **Weapons** — component objects added via `addWeapon()`; player fires indexed weapon, AI fires all
- **Particle pool** — `engine/systems/particlePool.js`, fixed slot count, presets: `explosion()`, `engineTrail()`
- **Location system** — `CONTENT.locations` is a unified flat table keyed by id; each entry has a `locationType` (`LOCATION_TYPE.STATION` or `LOCATION_TYPE.PLANET`). Stations store their entity descriptor on `entity`, planets store `backgroundData`. Helper `getLocationsByType(type)` in `data/dataRegistry.js` filters by type. Auto-discovered via `import.meta.glob` — no manual imports needed.
- **MAP format** — maps use a single flat `entities[]` array of pre-instantiated objects. `game.js` has one loop: `for (const entity of map.entities) { push to entities; if Ship, push to ships }`. Location manifests (e.g. `data/locations/tyr/pale/orbital/manifest.js`) export `{ entities[], zones[], background[] }` which maps spread.
- **Map data** — `data/maps/tyr.js` is the full production map; `data/maps/` holds all named maps (tyr, arena, blank); each exports `MAP`
- **Centralized stats** — `data/dataRegistry.js` holds equipment tables (ENGINES, WEAPONS, etc.) and `CONTENT` sub-tables (hulls, ships, conversations, derelicts, terrain, characters, modules, weapons, locations, history, factions). Three helpers: `registerData(table, entries)` for bulk equipment, `registerContent(type, id, entry)` for single content entries, `getLocationsByType(type)` to filter locations by `LOCATION_TYPE`. Content files self-register at import time. `data/index.js` uses `import.meta.glob` to auto-discover content — no manual imports needed. See `@SPEC.md` Data section for all current values. Directory layout:
  - **Global content:** `data/hulls/` (hull classes), `data/ships/player/` (player ships), `data/characters/player.js`, `data/modules/` (equipment), `data/maps/`, `data/lore/` (history + factions), `data/conversations/` (generic)
  - **Location content:** `data/locations/<system>/<body>/<node>/` — stations, ships, characters, derelicts, terrain, planets, conversations (auto-discovered)
  - `data/tuning.js` — global scalar constants. Each ship/weapon defines multiplier constants; never hardcode raw numbers.
- **Faction system** — `CONTENT.factions` (self-registered from `data/lore/factions/`). Root factions (no parent) are the 7 reputation-tracked factions; child factions inherit parent's reputation. Helpers in `data/factionHelpers.js`: `getFaction()`, `getFactionName()`, `getRootFaction()`, `areFactionsHostile()`, `areFactionsAllied()`, `getRelationToPlayer()`. NPC `relation` is derived per-tick in `game._updateRelations()` from faction reputation — content files should NOT hardcode `relation`. Provoked neutrals get `_relationOverride = true`. The old `FACTIONS`/`FACTION_LABELS`/`RIVALS` in `data/factions.js` are legacy — use `factionHelpers.js` for new code. See `@SPEC.md` Data section for faction table.
- **Thrust-to-weight** — `Ship.recalcTW()` derives speed/accel/turn/fuelEff purely from engine modules. Hull classes define only mass, durability, cargo, fuel tank, and armor — no inherent speed. Called event-based (module swap, cargo change, dock/undock). All NPC ships include engine modules.
- **Mount points** — each hull defines `MOUNT_POINTS[]` with `{ x, y, arc, size, slot? }`. Used for: module icons on hull, positional breach routing (arc-based), and install constraints (engine slots only accept engines). Module visuals: `engine/rendering/moduleVisuals.js`.
- **Module registry** — `engine/modules/registry.js` exports `createModuleById(id)`, which reads from `CONTENT.modules`. Used by ship configs and loot tables to instantiate modules by string ID.
- **Content registry** — `data/dataRegistry.js` exports `CONTENT` (type-keyed sub-objects) and `registerContent(type, id, entry)`. Content files call `registerContent()` at import time. Designer and editor read from `CONTENT.locations` (stations/planets), `CONTENT.derelicts`, `CONTENT.modules`, `CONTENT.weapons`, `CONTENT.factions`, etc. instead of hand-maintained registry arrays. Use `getLocationsByType(LOCATION_TYPE.STATION)` to get station entries.
- **UI overlays** — narrative panel (`#narrative-panel`, right 30%), ship panel (`#ship-panel`, left 30%), bottom HUD (`#hud-bottom`, 48px fixed bar). Docking sets `isDocked = true`, skipping the sim loop. Both panels use `stopPropagation` to prevent canvas input bleed. See `@SPEC.md` UX section for visual layout details.
- **Narrative system** — Disco Elysium-style conversation logs via `NarrativePanel` (`engine/ui/narrativePanel.js`). Conversations are async functions that `await log.choices(...)`, self-registered into `CONTENT.conversations`. Station-specific scripts in `data/locations/.../conversations/`; generic in `data/conversations/`. `game.storyFlags = {}` tracks NPC memory (session-only).
- **Color palette** — `engine/rendering/colors.js` exports all color constants; never use inline hex strings. See `@SPEC.md` UX section for full color tables.
- **CSS utility system** — `css/panel.css` defines CSS custom properties and typography/color utility classes. Never hardcode `px` font sizes in panel CSS — use `var()` references. See `@SPEC.md` UX section for full property/class tables.
- **Draw API** — `engine/rendering/draw.js` exports reusable canvas primitives:
  - **Immediate utilities** (take `ctx`): `polygon`, `polygonFill`, `polygonStroke`, `line`, `lines`, `disc`, `ring`, `trail`, `text`, `pulse`, `engineGlow`
  - **`Shape` class** — composable geometry with transform chaining (`.at()`, `.scaled()`, `.rotated()`). Factory methods: `Shape.rect()`, `Shape.chamferedRect()`, `Shape.cigar()`, `Shape.trapezoid()`, `Shape.wedge()`, `Shape.stadium()`, `Shape.cross()`, `Shape.ngon()`
  - **`DrawBatch` class** — deferred rendering that groups by style. Methods: `fillPoly`, `strokePoly`, `poly`, `line`, `disc`, `ring`, `rect`, `text`, then `flush()`
  - Always use Draw API over raw `ctx` calls. **Prefer Shape factories over raw point arrays** — add new factory methods rather than hand-placing points. Exception: complex hull shapes for armor arc rendering.

### Coordinate System

- Rotation 0 = pointing up (north, negative Y).
- World origin top-left; positive X right, positive Y down.

### Direction & Dimension Terminology

When the user says:
- **Width** — size along the X axis (left to right)
- **Height** — size along the Y axis (top to bottom)
- **Up / move up** — decrease Y (toward top of screen)
- **Down / move down** — increase Y (toward bottom of screen)
- **Left / Right** — decrease / increase X
- **On top / above** — drawn later (higher z-order, visually in front)
- **Underneath / below / behind** — drawn earlier (lower z-order, visually behind)

## Rules & Conventions

### Stats: Multiplier Pattern
Never hardcode raw stat numbers in ship/weapon constructors. All base values live in `data/*.js` (see Key Patterns above). Each ship/weapon file defines a multiplier (e.g. `HULL_MULT = 1.5`) and computes the final value as `BASE_HULL * HULL_MULT`. Global pacing knobs: `SPEED_FACTOR` and `PROJECTILE_SPEED_FACTOR` in `data/tuning.js`.

Ship classes use `this._initStats({ hull, weight, cargo, fuelMax, armorFront, armorSide, armorAft })` from `Ship` base to set hull stats. Movement stats (speed, acceleration, turn rate) and fuel efficiency are **purely engine-derived** via `recalcTW()` — hull classes never define them.

### Enums: Always Use `data/enums.js`
Never use raw strings for entity types, relations, conditions, loot types, arcs, or mount sizes. Import frozen enum objects (`ENTITY`, `RELATION`, `CONDITION`, `LOOT_TYPE`, `ARC`, `MOUNT_SIZE`, `MOUNT_SLOT`) from `data/enums.js`. The data lint script enforces this for `entityType` comparisons. If a new enum value is needed, add it to `data/enums.js` first.

### Colors: Always Use `engine/rendering/colors.js`
Never use inline hex strings anywhere in the codebase. Import named constants from `engine/rendering/colors.js`. If a new color is needed, add it there first.
